## TL;DR

**Root cause found.** The deployed `SlotService.ex5` is **stale**: it
was built from commit `30bdfa7` (v2.11) on 2026-07-23 19:22 UTC, but
three source-only commits landed afterwards that added `Print()`
debug output, the bind retry loop, the port change to 7780, and
the `bind 0.0.0.0` workaround. **The .ex5 was never rebuilt after
those source changes** — the debug prints we were looking for never
reached MT5 logs because they're not in the binary.

Rebuild + redeploy is the entire fix. No slot-side code change
required. The slot's REST/WS layer (13 endpoints, JWT auth,
`slot.stream` scope) is already wired and works; the broker
dispatch chain fails only because the MQL5 command server never
starts.

```
$ docker logs akroncloud-slot --since 30m | grep "MT5 service reported startup" | wc -l
~480    (one startup event every ~3-6s — service is in a tight restart loop)
```

The MT5 service is **starting**, calling `ConnectToSlot()` and
`SendStartupEvent()` (we see the startup events), then exiting
because... we still don't know exactly. Possibilities:

1. `bind()` returns `SOCKET_ERROR` with WSAEADDRINUSE (10048) —
   something else has 7780. (Unlikely — `ss` shows nothing.)
2. `bind()` returns `WSAEACCES` (10013) — wine's `ws2_32` denies
   bind to a non-localhost address. (Plausible — wine sometimes
   restricts to loopback unless the executable is "trusted".)
3. `bind()` succeeds, `listen()` fails.
4. The whole `StartCommandServer()` block is silently skipped by the
   stale binary because the **v2.11 binary predates the
   `StartCommandServer()` addition** — meaning v2.11 never had a
   command server at all, only the events socket. (This is
   consistent with the user's earlier observation that "output
   ~3KB más chico que v2.11b original" — v2.10/v2.11 without
   `StartCommandServer` would be ~3 KB smaller than a version
   that includes it.)

The fourth hypothesis is testable: if v2.11 doesn't include
`StartCommandServer()`, then **adding it now is the entire fix**,
not "make v2.11 work harder". A new `.ex5` built from the current
source would be ~94 KB (vs 90 KB for v2.10), consistent with the
+3952 byte diff observed between commits `78972e3` (53 KB) and
`30bdfa7` (94 KB).

## What's broken right now

The slot is honest about the failure:

```
{"ok":false,"reason":"MT5 command socket not connected"}
{"status":502,"code":"BROKER_DOWN","title":"Broker dispatch failed",
 "detail":"MT5 command socket not connected"}
```

The `Mt5CommandClient` (src/services/mt5-command-client.ts) tries
to `connect()` to `127.0.0.1:7780` every 2 s; on `ECONNREFUSED`
it logs at **debug** level only (`log.debug(...)`), so the failure
is invisible in normal logs. After a 1 s wait inside
`ensureConnected()`, the dispatch throws and the REST handler maps
that to `502 BROKER_DOWN`.

The fallback path in `dispatchCommand` (mt5-tcp-server.ts:208-227)
that would write commands to the events socket (`this.sock`) is
**unreachable**: `setCommandClient(mt5CmdClient)` in app.ts:103 is
always called, so `this.cmdClient` is non-null and the fallback is
dead code. Even if it were reached, MQL5 doesn't `recv()` on the
events socket today (it's write-only from the MQL5 side), so
fallback would silently drop commands anyway.

## What is NOT the fix

For posterity, the alternative options the previous session
considered are all unnecessary once we know the binary is stale:

- **Revert to events-socket-only design** (commands on 7778, no
  separate port) — would require MQL5 source change to `recv()`
  on `g_cmdSock` in `OnTimer()`. **Skip**: the v2.11 design is
  sound, the binary just doesn't have it.
- **Python relay on port 7780** writing to `MQL5/Files/slot-cmd.json`
  — would require a new Python service AND MQL5 source change to
  read the file. **Skip**: same reason, and it's strictly worse
  (extra hop, extra moving parts).
- **iptables REJECT fallback** for liveupdate — already in place
  via `scripts/suppress-liveupdate.sh` (3 layers: iptables +
  pkill + rm every 2s). The iptables layer logs a warning on
  the container because kernel capabilities don't include
  NET_ADMIN, but the pkill+rm layers work. **Already done; not
  the blocker.**
