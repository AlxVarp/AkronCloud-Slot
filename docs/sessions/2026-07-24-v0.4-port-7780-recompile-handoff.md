## Compile workflow (manual step for the human)

The Dockerfile (lines 350-358) documents the compile command:

```bash
xvfb-run -a wine /config/.wine/drive_c/users/abc/MetaTrader\
  5/MetaEditor64.exe /portable \
  /compile:"Z:\\config\.wine\drive_c\Program Files\MetaTrader 5\\MQL5\\Services\\SlotService.mq5"
```

But `metaeditor64.exe /compile:` is documented as **flaky in wine 11.0**
(Dockerfile line 350) — long timeouts, hung processes. The user has
been compiling this manually on a real Windows host or via MetaEditor's
GUI rather than the CLI. Either path works; the produced `.ex5` is the
same.

### Step-by-step

1. **Pull master locally**:
   ```bash
   git pull origin master
   # branch should be at 04e8db3 (or later)
   ```

2. **Open MetaEditor** in `MetaTrader 5` (on Windows or via wine GUI)

3. **Open `MQL5/Services/SlotService.mq5`** from the repo at this commit

4. **Compile** (`F7` or "Compile" button). Expected:
   - 0 errors
   - 5 warnings (the user has been seeing 5 consistently — these are
     benign and don't affect the binary)

5. **Save the output** as `mql5/SlotService.ex5`

6. **Report back the new MD5/SHA-256 and size** so we can verify
   the binary contains the debug + bind 0.0.0.0 changes:
   ```bash
   md5sum mql5/SlotService.ex5
   sha256sum mql5/SlotService.ex5
   stat --printf='size=%s\n' mql5/SlotService.ex5
   ```
   Expected size: ~95-97 KB (was 93 KB for v2.11, +2 KB for the
   debug strings and the second socket path).

## Expected WSA error codes (if 0.0.0.0 still doesn't bind)

If you compile from current master and port 7780 STILL doesn't bind,
we need to see the granular `bind()=N err=N` print (added by commit
6b7cbc7). But Print() output from MQL5 services is **not visible
from the sandbox** — it goes to MT5's "Services" tab in the GUI.
You'd need to open the MT5 GUI via `/desktop` and check the Experts
tab to see those prints.

The WSA error codes we'd expect if it still fails:

- `10048 WSAEADDRINUSE`: another process has port 7780. (Unlikely
  — `ss -lntp` from inside the container shows nothing on 7780.)
- `10013 WSAEACCES`: Wine denies the bind. (Possible; would need
  a different approach.)
- `10049 WSAEADDRNOTAVAIL`: address not available. (Possible with
  weird wine networking.)

## Fallback if recompile doesn't fix it

If after recompile port 7780 still doesn't bind, the realistic
paths are:

1. **Revert to events-socket commands** — modify `mql5/SlotService.mq5`
   to read commands from the events socket (already connected on
   7778) instead of binding a second TCP server on 7780. The slot's
   `dispatchCommand` fallback already writes commands to the events
   socket (mt5-tcp-server.ts:208-227) — but it's unreachable because
   `cmdClient` is always set. We'd need to either (a) skip cmdClient
   when it fails, or (b) have MQL5 listen on the events socket.

2. **Add a Python relay on port 7780** — a small `svc-mt5-cmd-relay`
   that listens on 7780 and writes commands to `MQL5/Files/slot-cmd.json`
   + reads responses from `MQL5/Files/slot-resp.jsonl`. MQL5 source
   change to read/write those files. ~50 lines of new code.

3. **Investigate wine 11.0's networking quirks** — e.g., try
   `winecfg` to see if Winsock is restricted, try different MT5
   builds, try running MT5 with `--single-instance` or `--no-update`.
