**Helper to add at top of class:**
```typescript
private async mt5Query<T = any>(
  action: string,
  payload: Record<string, any> = {},
): Promise<T> {
  const r = await this.tcp.dispatchCommand({
    type: 'command', action, payload,
    id: randomUUID(),
  });
  if (!r.ok) throw new Error(r.error ?? 'mt5_error');
  // r.result is a JSON STRING (MQL5 sends raw JSON text inside the
  // result field). Parse it.
  if (typeof r.result === 'string') return JSON.parse(r.result);
  if (typeof r.result === 'object' && r.result !== null) {
    return r.result as T;
  }
  return {} as T;
}
```

**Bonus fix:** existing `openTrade` and `closeTrade` cast `r.result`
directly as the typed object — but `r.result` is actually a string.
They work today only by accident (fall back to `pending-${id}`).
Fix them to use `JSON.parse(r.result)` like the helper above.

### 2. New file `src/api/trading.ts` — REST endpoints

All endpoints (except `/v1/health`, `/v1/state`, `/v1/sync`,
`/mobile`) require JWT via `Authorization: Bearer <token>`. Scopes:
- `slot:read` — read-only endpoints
- `slot:write` — place/modify/cancel

```

GET    /v1/symbols/:symbol         read   symbol detail
GET    /v1/quote?symbol=EURUSD     read   live bid/ask
GET    /v1/account                 read   leverage/currency/margin
GET    /v1/positions               read   open positions
POST   /v1/positions/:id/close    write  close (body: {volume?})
PATCH  /v1/positions/:id          write  modify SL/TP (body: {sl?,tp?})
GET    /v1/orders                  read   pending orders
POST   /v1/orders                  write  place (body: {side,instrument,qty,type,price?,sl?,tp?,client_order_id?})
GET    /v1/orders/:id              read   order detail
DELETE /v1/orders/:id              write  cancel
PATCH  /v1/orders/:id              write  modify SL/TP (body: {sl?,tp?})
GET    /v1/fills?from=&to=&limit=  read   deal history
```

Each handler does:
1. Verify JWT + scope (`read` for GET, `write` for POST/PATCH/DELETE)
2. Resolve accountRef (currently single-account — use the slot's configured account)
3. Call `deps.connector.<method>(...)` or `deps.connector.<method>(accountRef, ...)`
4. Return JSON result

## REST API shape — recommendation

Don't try to design a new schema. Map each endpoint 1:1 to a
SlotService action, returning the MQL5 response JSON almost as-is.
The cerebro or panel can translate to its preferred shape.

Example — POST /v1/orders:
```bash
curl -X POST http://45.151.122.104:7777/v1/orders \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"side":"buy","instrument":"EURUSD","qty":0.01,"type":"market"}'

# Response:
# {"ok":true,"order_id":"...","broker_order_id":"..."}
```

Example — GET /v1/positions:

```bash
#   "side":"buy","volume":0.01,"price_open":1.0832,...}]}
```

## Notes on the MQL5 protocol that the slot needs to know

- **Wire format:** newline-delimited JSON. `{"type":"response",
  "id":"<uuid>","ok":true|false,"result":<JSON STRING>}`. The
  `result` field is a JSON-encoded string, NOT a nested object.
- **Timeouts:** `CMD_TIMEOUT_MS` (default 5s) in
  `mt5-tcp-server.ts`. For multi-symbol queries like `symbols`
  this might be tight — bump via `SLOT_MT5_CMD_TIMEOUT_MS` env if needed.
- **One client at a time on TCP 7778.** When the cerebro connects
  for trading commands, the MQL5 publisher's connection gets
  replaced (the existing "replacing existing connection" log).
  That's fine for the cerebro use case (it commands when needed,
  doesn't stay connected). But during sustained WS streaming,
  the cerebro might want its own persistent connection — that's a
  future enhancement (deferred, see known limitations).

## Known limitations (carried forward)

- **One TCP client at a time on port 7778.** Trading commands
  conflict with the publisher's connection. The cerebro will need
  to use one connection at a time (REST calls are short-lived,
  WS is long-lived — design carefully).
- **MT5 IPC timeout on init** — the publisher still occasionally
  needs a restart after MT5 boot. The None-streak fix recovers
  automatically.
- **Latency p99 ~200ms from OpenHands → VPS** — irrelevant for
  real users who call from the cerebro (same datacenter).
