**Helper to add at top of class:**
```typescript
private async mt5Query<T = any>(
  action: string,
  payload: Record<string, any> = {},
): Promise<T> {
  const r = await this.tcp.dispatchCommand({
    type: 'command', action, payload,
    id: randomUUID(),
  });
  if (!r.ok) throw new Error(r.error ?? 'mt5_error');
  // r.result is a JSON STRING (MQL5 sends raw JSON text inside the
  // result field). Parse it.
  if (typeof r.result === 'string') return JSON.parse(r.result);
  if (typeof r.result === 'object' && r.result !== null) {
    return r.result as T;
  }
  return {} as T;
}
```

**Bonus fix:** existing `openTrade` and `closeTrade` cast `r.result`
directly as the typed object ‚Äî but `r.result` is actually a string.
They work today only by accident (fall back to `pending-${id}`).
Fix them to use `JSON.parse(r.result)` like the helper above.

## Notes on the MQL5 protocol that the slot needs to know

- **Wire format:** newline-delimited JSON. `{"type":"response",
  "id":"<uuid>","ok":true|false,"result":<JSON STRING>}`. The
  `result` field is a JSON-encoded string, NOT a nested object.
- **Timeouts:** `CMD_TIMEOUT_MS` (default 5s) in
  `mt5-tcp-server.ts`. For multi-symbol queries like `symbols`
  this might be tight ‚Äî bump via `SLOT_MT5_CMD_TIMEOUT_MS` env if needed.
- **One client at a time on TCP 7778.** When the cerebro connects
  for trading commands, the MQL5 publisher's connection gets
  replaced (the existing "replacing existing connection" log).
  That's fine for the cerebro use case (it commands when needed,
  doesn't stay connected). But during sustained WS streaming,
  the cerebro might want its own persistent connection ‚Äî that's a
  future enhancement (deferred, see known limitations).

## Known limitations (carried forward)

- **One TCP client at a time on port 7778.** Trading commands
  conflict with the publisher's connection. The cerebro will need
  to use one connection at a time (REST calls are short-lived,
  WS is long-lived ‚Äî design carefully).
- **MT5 IPC timeout on init** ‚Äî the publisher still occasionally
  needs a restart after MT5 boot. The None-streak fix recovers
  automatically.
- **Latency p99 ~200ms from OpenHands ‚Üí VPS** ‚Äî irrelevant for
  real users who call from the cerebro (same datacenter).

## What did NOT verify (and why)

### Single-TCP-client limitation on port 7778 (pre-existing)

The slot's MT5 TCP bridge accepts exactly one client at a time.
Two clients compete for it on the live VPS:

1. **SlotService.ex5** (MQL5) ‚Äî connects every ~30s, sends
   `startup` event, then waits for commands
2. **`mt5-account-publisher.py`** (Python, s6-supervised) ‚Äî
   publishes `account_status` events every 1.5s

The slot's `mt5-tcp-server.ts` accepts the first client and
**destroys** any existing connection when a new one arrives. The
result on the live VPS is that `this.sock` constantly flips
between MQL5 and Python. A `dispatchCommand` from the slot's REST
handler may:

- write to the Python publisher's connection ‚Üí no response from
  the publisher ‚Üí 5s `mt5_timeout`
- write to MQL5's connection but MQL5 is mid-reconnect ‚Üí
  socket destroyed before response ‚Üí `mt5_timeout` or
  `MT5 socket not connected`
- write to MQL5 in a stable window ‚Üí succeed (not observed in
  this session)

This is documented in the handoff ¬ßKnown limitations and in the
v0.3 stability-fix handoff ¬ßKnown limitations (third bullet). It
is not introduced by this PR.

### Python publisher `mt5.initialize()` IPC timeout (pre-existing, Finding C)

The Python publisher's `mt5.initialize()` returns `(-10005,
'IPC timeout')` repeatedly in this Wine 11.0 environment. The
publisher degrades to heartbeat-only mode (publishes
`logged_in:false, last_error:"mt5-init-timeout"`). The slot still
sees `loggedIn: true, balance: 9696.32` because:

1. login-detector (v53) publishes `{kind:'account',
   data:{logged_in:true}}` when the user logs into MT5 via
   `/mobile`
2. Some earlier MQL5 publish succeeded and cached the real
   balance/equity

Documented as Finding C in
`docs/sessions/2026-07-22-slot-service-v54-handoff.md`. Out of
scope for v0.4.

### Post-restart connector init issue (newly observed)

After every container restart, the `Mt5Connector.accounts` Map
is empty. The login-detector publishes `logged_in:true` events,
but `Mt5Connector.handleEvent` (mt5.ts:466) does
`const ref = this.firstRef(); if (!ref) return` ‚Äî silently
dropping the event because no accountRef is registered yet.

A `POST /v1/sync` (which triggers `validateAccount` ‚Üí
`connector.connect()`) re-registers the account in the Map. After
sync, the slot is ready for trading commands.

This is not new code in this PR, but it surfaces a wart: in
production, the slot should re-`connect()` every active account
on startup, not require a manual sync after every container
restart. Tracked as follow-up.
